# Phase B Plan v2 — Discord-Reviewed Content Pipeline

**Owner:** David Gaylor / NSMT
**Date:** 2026-05-21
**Status:** Draft v2 — incorporates Codex adversarial review findings
**Previous:** v1 in git history (commit before this rewrite)

## Goal (unchanged)

Articles generated by `nsmt-content-pipeline` post to Discord `#recap-pipeline` for human review **before** going live on thensmt.com. A reviewer's explicit approval action triggers the push to admin. Discord is the review surface; the durable state machine lives outside Discord.

## What changed from v1

| v1 (rejected) | v2 (this plan) |
|---|---|
| Discord forum tags ARE the state | KV ledger IS the state; Discord tags are display-only |
| `Approved` + absent `Published` triggers publish | Approval is an explicit action that mutates the ledger |
| "Check tag immediately before push" as lock | True concurrency guard via ledger CAS + cron-scoped lock |
| Markdown edits via PR | Markdown edits happen in ONE place (the .md file in repo); reviewer UX simplified |
| Worker falls back to default webhook on unknown target | Worker fails closed on unknown target |
| Bot needs Manage Threads + guild-wide Send Messages | Posting stays webhook-only; no bot perm expansion needed for MVP |
| Active-thread polling | Ledger-driven; doesn't depend on Discord thread state |
| Slug pattern undefined | Single canonical slug: `{slug-of-team-name}-recap-{YYYY-MM-DD}` |
| Admin duplicate check via grep of GET /blogs | Idempotency via ledger `admin_blog_id` field — never re-publish |

## Architecture

```
┌───────────────────────────────────────────────────────────────────┐
│ generate_content.py  (GH Actions, daily 8am ET)                   │
│  - ESPN scores → Claude → recap markdown                          │
│  - Writes drafts/{slug}.md (additive only, errors on overwrite)   │
│  - Commits + pushes to repo                                       │
│  - POST /draft-create to Cloudflare Worker                        │
│    payload: { slug, title, excerpt, md_url, team, category }      │
└───────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌───────────────────────────────────────────────────────────────────┐
│ Cloudflare Worker (nsmt-discord-proxy, enhanced)                  │
│  - Validates X-NSMT-Auth                                          │
│  - KV: if RECAPS:{slug} exists → return 409 (idempotency)         │
│  - KV: create ledger {status:'draft', slug, md_url, created}      │
│  - Discord webhook POST → forum thread w/ excerpt + md_url link   │
│  - Capture thread_id from Discord response, write back to KV      │
│  - Return 201 with ledger id                                      │
└───────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌───────────────────────────────────────────────────────────────────┐
│ Discord #recap-pipeline forum thread                              │
│  - Reviewer reads excerpt + clicks link to full markdown          │
│  - Discussion happens in thread comments                          │
│  - Edits happen by direct push to drafts/{slug}.md (or PR)        │
│  - Approval = explicit action (CLI or slash command, see below)   │
└───────────────────────────────────────────────────────────────────┘
                              │
                              ▼  (reviewer triggers approve)
┌───────────────────────────────────────────────────────────────────┐
│ Approval endpoint on Worker: POST /approve                        │
│  body: { slug, reviewer }                                         │
│  - Validates X-NSMT-Auth                                          │
│  - KV: read RECAPS:{slug}; require status == 'draft'              │
│  - KV: CAS status → 'approved' (rejects if status changed)        │
│  - Returns 200                                                    │
└───────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌───────────────────────────────────────────────────────────────────┐
│ Worker scheduled handler (Cloudflare Cron, every 5 min)           │
│  - KV.list({prefix: 'RECAPS:'})                                   │
│  - For each entry where status == 'approved':                     │
│    - CAS status → 'publishing' with 60s TTL (rejects if taken)    │
│    - Fetch md from md_url (GitHub raw)                            │
│    - markdown → HTML (using `marked` bundled in worker)           │
│    - POST to NSMT admin /admin/blogs with is_active=true          │
│    - On success: KV update {status:'published', admin_blog_id:X}  │
│    - On failure: KV update status back to 'approved' (retry)      │
│    - Post reply to Discord thread: "✅ Published <url>"           │
└───────────────────────────────────────────────────────────────────┘
```

## Detailed state machine (the ledger)

KV namespace `RECAPS`. Key pattern: `RECAPS:{slug}`. Value (JSON):

```json
{
  "slug": "washington-commanders-recap-2026-05-20",
  "title": "Commanders 28, Cowboys 21 | Washington Commanders Recap",
  "team": "Washington Commanders",
  "category": "pro",
  "md_url": "https://raw.githubusercontent.com/thensmt/nsmt-content-pipeline/main/drafts/washington-commanders-recap-2026-05-20.md",
  "thread_id": "1234567890",
  "status": "draft" | "approved" | "publishing" | "published" | "killed" | "failed",
  "admin_blog_id": null | "BLOG#abc-...",
  "live_url": null | "https://thensmt.com/blog/washington-commanders-recap-2026-05-20",
  "publishing_started_at": null | "2026-05-21T13:45:00Z",
  "publishing_attempts": 0,
  "last_error": null | "string",
  "reviewer": null | "david@nsmtsports.com",
  "created_at": "2026-05-21T13:00:00Z",
  "updated_at": "2026-05-21T13:42:00Z"
}
```

State transitions:
```
draft ──[/approve]──→ approved ──[cron CAS]──→ publishing ──[admin OK]──→ published
                          ▲                          │
                          └──[admin fail / retry]────┘
draft ──[/kill]────→ killed
```

Forbidden transitions enforced by CAS on every write. CAS is achieved with Cloudflare Workers KV's `cas` parameter (uses the metadata's `current_revision` token returned by `KV.getWithMetadata`).

## Components to build

### A. Cloudflare Worker (substantial enhancement)

Existing worker at `discord-proxy/src/index.js` is currently ~80 lines, one-job (forward to webhook). Phase B turns it into a small app:

- **New KV binding** `RECAPS` (added in wrangler.toml, created via `wrangler kv namespace create`)
- **Routes:**
  - `POST /` — existing webhook-forward behavior (keep for nsmt-form, nsmt-critique)
  - `POST /draft-create` — new (called by generate_content.py)
  - `POST /approve` — new (called by approval trigger)
  - `POST /kill` — new (kills a draft without publishing)
  - `GET /status/:slug` — new (read ledger for debugging)
- **Scheduled handler** `scheduled(event, env, ctx)` registered via wrangler.toml `[triggers] crons = ["*/5 * * * *"]`
- **`marked`** library bundled for markdown→HTML
- **Fail-closed routing:** existing webhook path rejects unknown `X-NSMT-Target` with 400

Estimated LOC: ~250 new lines on top of existing 80.

### B. `generate_content.py` modifications

- Read new env vars: `WORKER_BASE_URL`, `WORKER_SHARED_SECRET`, `GITHUB_RAW_BASE`
- Reconcile slug to canonical `{team-slug}-recap-{YYYY-MM-DD}` everywhere (currently inconsistent — see Codex P1.4)
- `save_local_draft`: error if file already exists (no silent overwrite). Add `--force` flag for explicit re-run.
- Replace direct admin POST with `POST {WORKER_BASE_URL}/draft-create`
- Remove `NSMT_USERNAME`/`NSMT_PASSWORD` from env requirements
- Add CLI flags: `--dry-run` (no Discord post, no commit), `--date YYYY-MM-DD`

### C. `daily-content.yml` GitHub Actions workflow

- Add `permissions: contents: write`
- Replace `NSMT_USERNAME`/`NSMT_PASSWORD` secrets with `WORKER_BASE_URL`, `WORKER_SHARED_SECRET`, `GITHUB_RAW_BASE`
- Add commit step:
  ```yaml
  - name: Commit & push drafts
    run: |
      git config user.name "github-actions[bot]"
      git config user.email "github-actions[bot]@users.noreply.github.com"
      git add drafts/
      if git diff --cached --quiet; then
        echo "No new drafts."
      else
        git commit -m "chore: daily recaps $(date -u +%Y-%m-%d)"
        git push
      fi
  ```
- Wrap pipeline + commit in a single job so partial failure is visible
- Concurrency guard:
  ```yaml
  concurrency:
    group: generate-content
    cancel-in-progress: false
  ```

### D. Approval trigger (MVP: CLI; Phase 2: Discord slash command)

MVP — `bin/approve.sh` script in the repo:
```bash
#!/bin/zsh
# Usage: bin/approve.sh <slug>
SLUG="$1"
[[ -z "$SLUG" ]] && { print -u2 "usage: approve.sh <slug>"; exit 1; }
curl -fsS -X POST "$WORKER_BASE_URL/approve" \
  -H "X-NSMT-Auth: $WORKER_SHARED_SECRET" \
  -H "Content-Type: application/json" \
  -d "{\"slug\":\"$SLUG\",\"reviewer\":\"$(git config user.email)\"}"
```

Phase 2 — registered Discord slash command `/approve` and `/kill`. Requires a real Discord application interaction endpoint (the worker can host the interaction handler since it's already public + has secrets). Slash commands are bot-token registered but interactions can be webhook-style if `INTERACTIONS_ENDPOINT_URL` is set in the Discord dev portal. Defer to Phase B.5.

### E. Markdown→HTML conversion in worker

Use `marked` (or `markdown-it`) bundled via wrangler. Minimal allowlist: `<p>`, `<em>`, `<strong>`, `<a>`, `<ul>`, `<li>`, `<blockquote>`. Strip everything else for safety since the admin frontend renders the HTML inline without server-side sanitization (per FOR_QUINCY.md instructions to use React's raw-HTML render path). Our worker is the LAST sanitization point before the article hits the admin DB, so be strict here.

### F. Slug reconciliation

Current code has THREE patterns (Codex P1.4):
1. `save_local_draft`: `drafts/{date}-{team-slug}.md`
2. `save_to_nsmt`: slug = `{team-slug}-recap-{date}`
3. Plan v1 assumed: `{date}-{slug}.md`

**Canonical from v2 onward:**
- Slug = `{team-slug}-recap-{date}` (used by admin already)
- File path = `drafts/{slug}.md`
- KV key = `RECAPS:{slug}`
- Discord thread name = first 80 chars of `title`, NOT slug (human-readable)
- Thread→ledger lookup via thread_id stored in KV, NOT via thread name parse

This eliminates the parse-thread-name fragility from v1.

## How v2 addresses each Codex finding

| Codex finding | v2 mitigation |
|---|---|
| **P0.1** Tags are an array, not enum | Ledger status is single-valued; tags are display-only |
| **P0.2** "Check tag before push" not a lock | KV CAS on status='publishing' transition; only one writer succeeds |
| **P0.3** Admin has no slug filter | Ledger stores admin_blog_id; we never need to ask admin "does X exist" |
| **P1.4** Slug patterns inconsistent | Reconciled to single canonical form (section F) |
| **P1.5** Git+Discord non-atomic | Worker handles both KV write AND Discord post in one request; rollback on Discord failure |
| **P1.6** save_local_draft overwrites | Now errors on existing file unless `--force` |
| **P1.7** Worker falls back on unknown target | Fail closed: return 400 |
| **P1.8** Bot perms too broad | MVP uses webhook only (no bot needed); Phase B.5 adds slash commands with INTERACTIONS_ENDPOINT_URL pattern (still no bot Manage Threads) |
| **P1.9** Active-thread polling misses archived | Doesn't matter — we poll the ledger, not Discord |
| **P2.10** Reviewer UX friction | Reduced: read excerpt in Discord, full md in single GitHub view, approve via one CLI command. No more "PR → merge → return to Discord → re-tag" |
| **P2.11** Markdown/HTML contract | Explicit `marked` conversion in worker, allowlisted tags |
| **P2.12** Model name wrong | Acknowledged — not in this plan's scope |

## New risk register

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| KV consistency (eventual reads) | Med | Low | CAS handles race; eventual reads on `/status` acceptable |
| `marked` bundle size > worker limit (1MB) | Low | Med | Use `marked` (~50KB minified) not heavier alternatives |
| Cron handler timeout (10s default per request) | Med | Med | Process at most 5 drafts/run; rest wait for next cron |
| GH Actions `contents: write` lets workflow tamper | Low | Low | Only this workflow has the perm; only modifies `drafts/` |
| Concurrent generate runs (manual + cron same day) | Low | Low | Idempotency via 409 on existing slug; reruns are safe |
| Discord webhook deleted | Low | High | `/draft-create` errors loudly; reviewer manually triages |
| Reviewer edits .md after approve, before publish | Med | High | Cron uses md_url at publish time, so edits ARE captured; document this is expected |
| Reviewer edits .md after publish | High | Low | Plan won't re-publish; reviewer must use admin to update live article |
| KV value > 25MB | Negligible | N/A | Single ledger value is <1KB |
| Worker scheduled handler runs multiple times concurrently | Low | High | Cloudflare guarantees single-instance per cron schedule per worker |

## Acceptance criteria

- Generate run creates: drafts/{slug}.md, KV entry with status='draft', Discord thread with link
- Re-running generate for same date is idempotent (409 on second call)
- Approve via CLI flips KV status='approved'
- Within 5 min, cron picks it up, status→publishing→published, admin has the article (is_active=1), Discord thread gets a "Published" reply
- Worker rejects: unknown X-NSMT-Target (400), missing X-NSMT-Auth (401), unknown slug on /approve (404), already-approved on /approve (409)
- Killing a draft prevents publishing; rerunning generate for same slug is rejected
- Concurrent cron runs do not double-publish (CAS test)
